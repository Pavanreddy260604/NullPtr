c) What are the different approaches to Virtualization?

Approaches to Virtualization

Ø Protection rings are used to isolate the OS from untrusted user applications.

Ø In protection ring architecture, the rings are arranged in hierarchical order from Ring 0 to Ring 3.

Ø Ring 0 → contains the most privileged programs.

Ø Ring 3 → contains the least privileged programs.

Ø Highly trusted OS instructions run in Ring 0 and have unrestricted access to physical resources.

Ø Ring 3 contains untrusted user applications and has restricted access to resources.

Ø Ring 1 and Ring 2 are allotted for device drivers.

Ø Protection ring architecture restricts misuse of resources and malicious behavior of user programs.

Ø Depending on the type of virtualization, the hypervisor and guest OS will run at different privilege levels.

Ø Normally, the hypervisor runs at Ring 0 (most privileged).

Ø The guest OS runs at a lower privilege than the hypervisor.

The three types of approaches are:

Ø Full virtualization

Ø Para virtualization

Ø Hardware-assisted virtualization

 

 

Full Virtualization

Ø The guest OS is fully abstracted from the real hardware.

Ø The virtual machine manager (VMM) or hypervisor separates the guest OS from the infrastructure.

Ø The guest OS does not know it is virtualized – it thinks it runs on real hardware.

Ø Hypervisor (VMM) runs at Ring 0 (highest privilege).

Ø Guest OS runs at Ring 1 (lower privilege than hypervisor).

Ø User applications run at Ring 3 (least privilege).

Ø The OS cannot talk directly to physical hardware – it must go through the hypervisor.

Uses two techniques:

Ø Binary translation → translates guest OS instructions into safe instructions for virtual hardware.

Ø Direct execution → allows user applications to directly use resources without changes.

Pros

Ø This approach provides the best isolation and security for the VMs.

Ø Different OSs can run simultaneously.

Ø The virtual guest OS can be easily migrated to work in native

     hardware.

Ø It is easy to install and use and does not require any change in the  guest OS.

 

 

Cons

Ø Binary translation is an additional, overhead, and it reduces the

Ø overall system performance.

Ø There is a need for correct combination of hardware and software

 

Paravirtualization

Ø Also called partial virtualization or OS-assisted virtualization.

Ø Provides partial simulation of the underlying infrastructure.

Ø Main difference from full virtualization:

Ø In full virtualization, the guest OS does not know it is virtualized.

Ø In paravirtualization, the guest OS knows it is in a virtualized environment.

Ø Instead of binary translation, hypercalls are used.

Ø Hypercalls = like system calls, allow direct communication between guest OS and hypervisor.

Ø This direct communication improves performance and efficiency.

Ø In full virtualization → guest OS runs unmodified.

Ø In paravirtualization → guest OS must be modified to replace certain instructions with hypercalls.

Architecture:

Ø Modified guest OS runs at Ring 0.

Ø User applications run at Ring 3.

Ø Guest OS at Ring 0 can communicate directly with the hypervisor via hypercalls.

Ø Like full virtualization, user applications can access hardware directly.

Pros

Ø It eliminates the additional overhead of binary translation and hence improves the overall system efficiency and performance.

Ø  It is easier to implement than full virtualization as there is no need

         for special hardware.

Cons

Ø There is an overhead of guest OS kernel modification.

Ø The modified guest OS cannot be migrated to run on physical hardware.

Ø VMs suffer from lack of backward compatibility and are difficult to   migrate to other hosts.

 

Hardware-Assisted Virtualization

Ø In full and paravirtualization, there is overhead due to binary translation or modifying the guest OS.

Ø In this approach, hardware vendors (Intel, AMD) provide built-in support for virtualization → reduces overhead.

Ø Examples: Intel VT-x, AMD-V

Ø VT-x uses virtual machine control structures.

Ø AMD-V uses virtual machine control blocks.

Ø VMM (hypervisor) has the highest privilege (root privilege) even below Ring 0.

Ø OS runs at Ring 0, and user applications at Ring 3.

Ø In this method, guest OS and applications share the same non-root privilege level.

Ø Removes the need for binary translation or OS modification.

Ø OS requests go directly to the hypervisor (trapped without translation).

Ø User requests are executed directly, just like in other virtualization approaches.

Pros

Ø It reduces the additional overhead of binary translation  in full

    virtualization.

Ø It eliminates the guest OS modification in paravirtualization.

Cons

Ø Only new-generation processors have these capabilities. All

     x86/x86_64 processors do not support hardware-assisted

     virtualization features.

Ø More number of VM traps result in high CPU overhead, limited  scalability, and less efficiency in server consolidation.

c) Explain in detail about any 3 virtualization opportunities?

Virtualization Opportunities

Ø Virtualization is the process of abstracting physical resources into a pool of virtual resources.

Ø These virtual resources can then be allocated to Virtual Machines (VMs).

The different resources that can be virtualized include: Memory, Processors, Storage, Network, Data, and Application.

Processor Virtualization

Ø Processor virtualization lets VMs share virtual processors.

Ø These virtual processors are abstracted from physical processors.

Ø The virtualization layer (hypervisor) creates the pool of virtual processors.

Ø Processor virtualization can be done using a single hardware or distributed servers.

 

 

Memory Virtualization

Ø A resource virtualization technique that gives virtual main memory to VMs.

Ø Physical main memory is mapped to virtual memory, like in OS virtual memory.

Ø Main idea: map virtual page numbers to physical page numbers.

Ø All modern x86 processors support memory virtualization.

Ø Can also be done using hypervisor software.

Ø In virtualized data centers, unused memory from servers is combined into a virtual memory pool for VMs.

 

 

Storage Virtualization

Ø A type of resource virtualization where many physical disks are combined into a pool of virtual storage for VMs.

Ø This virtualized storage is called logical storage.

Ø Mainly used for backup or replica of data stored on VMs.

Ø Supports high availability of data.

Ø Can be achieved through hypervisors.

Ø Makes efficient use of physical storage.

Ø Advanced techniques: Storage Area Networks (SAN) and Network-Attached Storage (NAS).

c) List the best practices to keep the Hypervisor secured?

Most of the attacks on the hypervisor are through the host OS or the guest OS. So, the OS should be protected from the malicious attacker or the hypervisors should be protected from the com­promised OSs. There are several best practices to keep the hypervisor secured:

 

·     Update the hypervisor software and the host OS regularly.

·     Disconnect the unused physical resources from the host system or hypervisor.

·     Enable least privilege to the hypervisor and guest OS to avoid the attacks through unauthorized access.

·     Deploy the monitoring tools in the hypervisor to detect/prevent malicious activities.

·     Strong guest isolation.

·     Employ mandatory access control policies.

 

d)How can we prove that Virtualization and Cloud computing are different?

Many users of current IT solutions consider the technologies virtualization and cloud computing as the same. But both technologies are actually different, or in other words, we can say virtualization is not cloud computing. We can prove this claim with the following parameters:

1. Type of service: Generally, virtualization offers more infrastructure services rather than platform and application services. But cloud computing offers all infrastructure (IaaS), platform (PaaS), and soft­ware (SaaS) services.

2. Service delivery: The service delivery in cloud computing is on-demand and allows the end users to use the cloud services as per the need. But virtualization is not made for on-demand services.

3. Service provisioning: In cloud computing, automated and self-service provisioning is possible for the end users, whereas in virtualization, it is not possible and a lot of manual work is required from the provid­ers or system administrator to provide services to the end users.

4. Service orchestration: Cloud computing allows the service orchestra­tion and service composition to meet end user requirements. Some providers are also providing automated service orchestration to the end users. But in virtualization, orchestrating different service to get composite services is not possible.

5. Elasticity: One of the important characteristics that differentiate cloud computing from virtualization is elasticity. In cloud comput­ing, we can add or remove the infrastructure dynamically according to the need, and adding or removing the infrastructure is automatic. But virtualization fails to provide elasticity as stopping and starting a VM is manual and is also difficult.

6. Targeted audience: The targeted audience of these two technologies is also different. Cloud computing targets the service providers for high resource utilization and improved ROI. At the same time, it also facilitates the end users to save money by using on-demand ser­vices. In the case of virtualization, the targeted audience is only the service providers or IT owners, not the end users.

c) Explain in detail about Cloud Haskell?

Cloud Haskell: Functional Programming

Ø Cloud Haskell is built on the Haskell functional programming language.

Ø Functional programming works like mathematical functions — the output depends only on the input.

Ø It uses immutable components, meaning their state cannot be changed after creation.

Ø These immutable components are the foundation of Cloud Haskell.

Ø Cloud Haskell is a domain-specific language (designed for a specific purpose).

Programming Model

Ø The programming model is based on a message-passing interface.

Ø It is suitable for reliable and real-time applications.

Ø It follows a concurrent programming style — many processes run at the same time.

Ø The cloud has many users and applications, so concurrency helps handle many requests efficiently.

Ø Each application is independent, and data are isolated — one cannot access another’s data.

Ø Communication happens only through message passing.

Ø A cost model is used to measure data movement between processes.

Main Features and Advantages

Ø Uses Erlang’s fault tolerance concept.

Ø Pure functional language — all data are immutable by default.

Ø Functions are idempotent, meaning they can restart automatically if stopped due to an error, without needing outside help.

Fault Tolerance

Ø Failures can happen often in cloud systems as the number of users and nodes increases.

Ø Normally, systems restart completely after failure — this is inefficient for large clouds.

Ø Cloud Haskell uses Erlang’s method, allowing individual functions to restart separately without affecting others.

Ø Components are immutable and called pure functions.

Cost Model and Serialization

Ø Cloud Haskell introduces a new cost model (not present in Haskell).

Ø It calculates communication cost between processes.

Ø The model does not use shared memory.

Ø Each process is isolated, so if one fails, others continue (ensuring fault tolerance).

Ø Adds new features like serialization, allowing distributed code to run on remote systems automatically.

Ø This process is fully abstract — the programmer doesn’t have to handle it manually.

Key features

Ø Fault tolerant

Ø Domain-specific language

Ø Uses concurrent programming paradigm

Ø No shared memory

Ø Idempotent

Ø Purely functional

d) Write short notes on Map Reduce?

MapReduce is a parallel programming model developed by Google.

Ø It has a specific impact on the area of parallel computing and cloud computing.

Ø It uses a parallel programming paradigm for the computation of a large amount of data.

Ø The data are partitioned or segregated into several pieces.

Ø These pieces are processed in parallel.

Ø The results are then combined to generate a single result.

Ø These processes are called map and reduce, respectively.

Ø The PaaS called Aneka also uses MapReduce with the .NET platform .

Ø The first step involves segregation of large data and sending it to the map function.

Ø The map function performs the map operation on data.

Ø Then, the data are sent for the reduce operation, where the actual results are obtained.

Map Function

Ø The large data are traced and segregated as key/value pairs.

Ø The map function accepts the key/value pair and returns an intermediate key/value pair.

Ø Usually, a map function works on a single document.

Formula for operation:

Map(Key₁, Value₁) → List(Key₂, Value₂)

Ø The list of key/value pairs obtained is subsequently sent to the reduce function.

Reduce Function

Ø After the map function, a sort and merge operation is applied on the key/value pairs.

Ø Usually, the reduce function works on a single word.

Ø The input consists of distinct key/value pairs, and it returns a group with the same key/value pair.

Formula for operation:

Reduce(Key₂, List(Value₂)) → List(Value₃)

Ø The resultant values give the final set of results.

Ø Key Features of MapReduce-

Supports parallel programming,

Fast,                                           

Can handle a large amount of data

Example

Ø The most popular example is counting the words in a file.

Ø In the first step, words (keys) are assigned values (counts).

Ø The word represents the key, and the count represents the value.

Ø Whenever a word is encountered, a key/value pair is generated.

Ø This set of <word, count> pairs from the map function is sent to the reduce function.

Ø The result is sorted, and the reduce function combines all words with the same key.

Ø The count (value) is added for each pair obtained.

Ø The final result gives each word with its count.

c) Explain in detail about the suitability of Saas?

Suitability of SaaS

SaaS is cost-effective and widely used by individuals and organizations, including large enterprises. However, it is not suitable for all types of applications.

SaaS May Not Be Suitable

Ø For real-time applications that need very fast data processing.

Ø When an organization has highly confidential data and doesn’t want to store it outside their system.

Ø When existing on-premise software already meets the organization’s needs.

SaaS Is the Best Option

Ø When users need on-demand software instead of buying full licenses.

Ø For start-ups that cannot afford expensive licensed software.

Ø For applications that must be accessible on mobile devices or thin clients.

Ø For applications with unpredictable or changing workloads.

Benefits of SaaS

SaaS gives customers a low-cost, on-demand, and easy way to use software without needing to buy or install it.
It is also simple to adopt and connect with other software.

1. Pay per Use

Ø Users pay only for what they use.

Ø SaaS works on a subscription basis — customers can start or stop anytime.

Ø In traditional software, users pay the full price even if they use it rarely.

2. Zero Infrastructure

Ø No need to buy hardware, operating systems, or updates.

Ø SaaS reduces capital costs, as everything runs on the provider’s infrastructure.

3. Ease of Access

Ø Needs only a web browser and internet connection.

Ø The responsive interface adjusts automatically to any device (PC, tablet, phone).

4. Automated Updates

Ø Service provider handles all updates automatically.

Ø Users always get the latest version without manual installation.

5. Composite Services

Ø SaaS can integrate with other software or cloud services using APIs.

Ø This helps create combined or extended services easily.

6. Dynamic Scaling

Ø SaaS can handle changing or unpredictable user loads.

Ø Uses dynamic load balancing to manage extra demand without slowing down.

7. Green IT Solutions

Ø SaaS supports eco-friendly computing.

Ø Because resources are shared (multi-tenant), it uses less energy and hardware.

Ø High resource use means less waste and lower power consumption.

c) Discuss the challenges that make the SaaS development difficult?

The challenges that make SaaS development difficult are:

1. Multitenancy

Ø Multitenancy means one application serves many customers (tenants) using a shared instance.

Ø It improves resource utilization.

Ø Can be implemented at infrastructure, platform, database, or application levels.

Ø In databases, it can be:

Ø Separate databases per tenant

Ø Shared database with separate schemas

Ø Shared database and shared schema

Ø Developers must choose the right level based on customer needs.

Ø Using PaaS can reduce the complexity of building multitenant systems.

2. Security

Ø Data security is a big challenge because tenants share the same environment.

Ø There is a risk of data breaches between tenants.

Ø Enterprises are often hesitant to adopt SaaS for this reason.

Ø Developers must isolate tenant data and use:

Ø Encryption

Ø Authentication & Authorization

Ø Access control

Ø Auditing

Ø Intrusion detection/prevention systems (IDS/IPS)

3. Scalability

Ø SaaS applications must handle extra or unpredictable loads smoothly.

Ø Developers should design scalable architectures using:

Ø Vertical scaling (add more power to existing servers)

Ø Horizontal scaling (add more servers)

Ø Load balancing

Ø IaaS and PaaS help improve scalability.

 

 

4. Availability

Ø SaaS must ensure 99.99% uptime for users.

Ø Use backup, replication, and disaster recovery mechanisms.

Ø Distributed NoSQL databases support automatic replication and failover, improving reliability.

5. Usability

Ø SaaS must support multiple devices (laptops, mobiles, tablets).

Ø Developers should design a responsive web interface (UI) that adapts to all screens.

Ø Each tenant’s customization settings must be saved properly for each device.

6. Self-Service Sign-Up

Ø SaaS must allow users to sign up instantly — no admin approval needed.

Ø Users should also be able to unsubscribe anytime.

Ø The application should become available immediately after registration.

7. Automated Billing

Ø SaaS must include automatic billing and usage tracking.

Ø Should prevent overcharging after users disconnect.

Ø Users should see total usage and per-service billing details.

8. Nondisruptive Updates

Ø Updates should not interrupt users or cause downtime.

Ø Must avoid SLA (Service Level Agreement) violations.

Ø Updates should be scheduled and smooth without affecting normal use.

9. Service Integration

Ø SaaS should easily integrate with third-party services using APIs.

Ø Use Service-Oriented Architecture (SOA) for better integration.

Ø Integration helps automate some tasks and improve functionality.

10. Vendor Lock-In

Ø Vendor lock-in happens when it’s hard to move from one cloud provider to another.

Ø Caused by different standards used by providers.

Ø Developers should choose IaaS and PaaS that are interoperable with other platforms to avoid lock-in.
